# Product Sense

## Product Goals

- Let a developer observe the current state of a harness work item without repeatedly reopening source files.
- Keep the TUI synchronized with active edits so it can stay open while another terminal is used for implementation.
- Reduce friction when answering questions such as:
  - Which phases exist in this plan?
  - Which requirements are validated, pending, or missing proof?
  - What do the PRD, FDD, and plan currently say?

## Primary User

- A harness-oriented engineer driving implementation from one terminal while monitoring a work item in another.

## Product Principles

- Local-first over service-heavy.
- Fast state inspection over authoring.
- Reliable refresh behavior over premature feature breadth.
